1. Fluxul de funcționare, cap-coadă

Diagrama de mai jos arată traseul complet al informației: de la datele de intrare, prin interfața HTML și API-ul Python, până la motorul de calcul și indicatorii executive folosiți în decizie.

Pas 1 · Input

Date de intrare

Tot ce alimentează modelul de estimare.

  • Date GIS și SAP PM
  • Fișier Excel importat manual
  • Echipe pe grup: GIS, RASR, FOL
  • Parametri: vMin, vMax, ore/săpt, săpt/an, target, factorC
Pas 2 · UI

Interfață HTML

Partea vizibilă pentru utilizator și management.

  • Afișează KPI executive și tabele
  • Permite modificarea parametrilor
  • Rulează simulări what-if
  • Trimite date spre backend prin API
Pas 3 · API

FastAPI / Python

Stratul care primește, validează și orchestrează calculele.

  • Endpointuri de calcul și import Excel
  • Validare date și reguli business
  • Protecție logică împotriva erorilor
  • Returnează rezultate consolidate
Pas 4 · Engine

Motor de calcul FTE

Aici se transformă resursele și volumul în timp estimat.

  • Capacitate anuală de lucru
  • ETA pe grup și ETA total
  • Scenarii worst / mediu / best
  • Identificare grupuri blocate
Pas 5 · Output

Indicatori pentru management

Rezultatele care se văd în dashboard și susțin decizia.

  • % corelat și km rămași
  • Grupuri în target / în risc / blocate
  • ETA estimat până la finalizare
  • Impactul suplimentării de resurse
Pas 6 · Decizie

Acțiune managerială

Locul în care informația devine decizie.

  • Unde suplimentezi FTE
  • Ce grupuri prioritezi
  • Ce target este realist
  • Care este impactul operațional

2. Conexiunile dintre HTML, API, repository și deploy

Dashboard-ul HTML este interfața din browser. El citește și afișează datele, dar pentru calculele oficiale apelează API-ul Python. Codul sursă trăiește în GitHub, iar Render publică aplicația live după procesul de aprobare și deploy.

HTML / Frontend

Vizualizează KPI-uri, tabele și simulări. Aici se fac modificările de parametri și interacțiunea cu utilizatorul.

Bridge / Request

Frontend-ul trimite către API datele despre grupuri, parametri și scenarii. Practic, este puntea dintre ecran și motorul de calcul.

API FastAPI

Primește datele, le validează și cheamă funcțiile Python care aplică regulile de business și formulele de estimare.

GitHub + Render

Orice update intră în repository, trece prin Pull Request și Merge, apoi Render face deploy și publică noua versiune live.

Utilizator → Dashboard HTML → API FastAPI → Motor Python → KPI / ETA / Scenarii → Decizie
Repository GitHub → Pull Request → Merge → Render Deploy → Dashboard Live

3. Principiul de estimare FTE

Modelul răspunde la întrebarea: Cu echipele actuale și productivitatea actuală, în cât timp terminăm corelarea GIS ↔ SAP PM?

Nr. echipe reale = min(GIS, RASR)
Ore/an = ore/săpt × săpt/an × (echipe + FOL × % contribuție)
Rată medie = (vMin + vMax) / 2
Ore necesare = km / rată medie
Luni ETA = (Ore necesare / Ore disponibile) × 12 × factorC

factorC este multiplicatorul de realitate operațională. El ajustează estimarea matematică ideală cu complexitatea din teren, blocajele și variația procesului.

4. Ce livrează concret managementului

Estimare de timp

Arată în câte luni se finalizează corelarea la nivel total și pe fiecare grup operațional.

Identificare blocaje

Evidențiază grupurile care nu au capacitate reală sau care ies din target.

Simulare decizională

Permite testarea impactului dacă adaugi oameni, schimbi parametri sau repoziționezi resurse.

Mesaj executiv Kaizen Dashboard nu este doar un tablou de bord. Este un instrument de transformare a volumului de lucru și a resurselor disponibile în timp estimat, risc și opțiuni de decizie privind FTE-ul. Cu alte cuvinte: traduce datele operaționale în decizii manageriale.
